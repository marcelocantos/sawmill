## Phase 1: Code Quality

### 1.1 Architecture and Design

**Severity: Info** -- The architecture is clean and well-layered. The codebase follows a clear separation of concerns:

- `forest.rs` -- parsing and file management
- `transform.rs` -- match/act engine
- `rewrite.rs` -- range-based source rewriting
- `codegen.rs` -- JavaScript runtime for code generation
- `store.rs` -- SQLite persistence
- `model.rs` -- ties everything together
- `mcp.rs` -- MCP tool definitions
- `adapters/` -- language-specific Tree-sitter queries

Module boundaries are respected. The `LanguageAdapter` trait provides a clean abstraction for language-specific behavior. The dependency flow is unidirectional with no circular dependencies.

One structural concern: `src/mcp.rs` at 2021 lines is the largest file by far and handles all tool definitions plus helper functions. It could benefit from splitting into sub-modules (e.g., `mcp/tools.rs`, `mcp/helpers.rs`).

### 1.2 Code Correctness

**Finding [3]** (Medium) -- `src/lsp.rs:399-408`: `try_read_message` has a broken timeout implementation. It checks `start.elapsed() < timeout` immediately after creating `start`, which will always be true (elapsed is ~0). This means it always attempts a blocking read regardless of the timeout parameter. The comment acknowledges this: "Simplified: attempt a blocking read with a timeout via thread. For production, use non-blocking I/O."

**Finding [4]** (Medium) -- `src/mcp.rs:1601-1603`: `check_conventions_on_changes` runs convention checks against the current (pre-change) forest rather than the post-change state. The code includes a comment acknowledging this limitation. Convention violations on the new code won't be caught until the next check.

**Finding [5]** (Medium) -- `src/forest.rs:110`: `undo_from_backups` uses `to_string_lossy().trim_end_matches(".sawmill.bak")` for path derivation. This is fragile -- it operates on the string representation of the path rather than using proper path extension stripping. The original approach (`with_extension("")`) was correctly identified as insufficient (only strips one extension level), but the string-based fix could misbehave with unusual paths containing ".sawmill.bak" as a substring.

**Finding [8]** (Low) -- `src/codegen.rs:144-155`: `LspProxy` uses `unsafe impl Send` and raw pointer dereferencing to pass the `LspManager` into the JS context. The safety comment is present and the usage is single-threaded, but this could be replaced with a `RefCell`-based approach or a proper message-passing pattern to eliminate the `unsafe`.

All Mutex usage is `.lock().unwrap()` -- acceptable for a single-threaded MCP server where mutex poisoning indicates a real bug, but would panic on poisoned mutexes rather than recovering.

### 1.3 Code Hygiene

**Finding [1]** (Medium) -- `cargo fmt --check` fails: `src/mcp.rs:1196` has a formatting discrepancy in `get_agent_prompt`. This will fail CI (the CI workflow runs `cargo fmt --check`).

**Finding [2]** (Medium) -- 2 clippy warnings: collapsible `if` statements at `src/mcp.rs:1208` and `src/mcp.rs:1226`. CI runs `cargo clippy -- -D warnings`, so these will also fail CI.

**Finding [6]** (Low) -- `src/watcher.rs:110` uses `eprintln!` for error output. The project doesn't currently use a structured logging crate, but for an MCP server communicating over stdio, `eprintln!` output could interfere with the JSON-RPC protocol. Consider either suppressing or routing to a file-based log.

No TODO/FIXME/HACK markers found in source code. No dead code concerns (the crate intentionally uses `#![allow(dead_code)]` for planned features, which is documented). Naming is consistent and idiomatic Rust throughout.

## Phase 2: Security

### 2.1 Input Handling

The JS execution engine (QuickJS) has reasonable limits: 2GB memory limit, 8MB stack. The JS transform functions are user-supplied but execute in a sandboxed QuickJS context without filesystem or network access.

SQL queries in `store.rs` use parameterised queries throughout -- no SQL injection vectors.

The `format_source` function in `rewrite.rs` shells out to external formatters (rustfmt, ruff, gofmt, etc.) but only passes source via stdin/stdout -- no shell injection risk.

### 2.2 Secrets and Credentials

No hardcoded secrets, API keys, tokens, or credentials found in source code or configuration files. `.env` files are properly listed in `.gitignore`.

### 2.3 Cryptography

No cryptographic operations in the codebase. `blake3` is used for content hashing (non-security purpose) -- appropriate.

### 2.4 Access and Exposure

The MCP server communicates over stdio only (not TCP/HTTP), so there is no network exposure surface. File operations are scoped to the directory the user points `parse` at.

## Phase 4: Performance

No significant performance concerns for a project of this size and nature.

- The `sync()` method in `model.rs:136` uses `changed_paths.contains(&p)` (linear scan) for dedup -- fine for typical file counts but could use a `HashSet` for very large repositories.
- File parsing is parallelisable via `rayon` (listed as a dependency) but currently runs sequentially in `Forest::from_path`. This is a reasonable starting point -- parallelism can be added when it becomes a bottleneck.
- QuickJS runtime is created fresh for each JS transform/codegen call. For repeated small transforms this has some overhead, but the alternative (keeping a persistent runtime) would add complexity for a marginal gain.
